# Android Proxy Dashboard Design

## Summary

Build a local Node.js web application that acts as a MITM proxy for Android app traffic routed through a desktop machine. The MVP must capture HTTP and HTTPS traffic, decrypt HTTPS when the Android device trusts the local CA, store sessions, and present request and response headers and bodies in a browser-based dashboard.

This tool is intentionally narrower than Charles or mitmproxy. It targets a single-user local workflow for inspecting API traffic from an Android device connected through a manually configured Wi-Fi proxy. The first release optimizes for observability and reliability, not protocol breadth or traffic manipulation.

## Goals

- Capture HTTP and HTTPS traffic from an Android device using the desktop machine as an explicit proxy.
- Decrypt HTTPS traffic when the device trusts the generated local CA.
- Store each request and response as a structured session with timing and error metadata.
- Provide a browser UI for browsing, filtering, and inspecting sessions.
- Expose setup and diagnostics information directly in the UI so Android configuration failures are visible.

## Non-Goals

- Bypassing SSL pinning.
- iPhone support.
- Rewrite, mock, breakpoint, or resend features.
- Enterprise multi-user support.
- Deep protocol coverage beyond the MVP HTTP/HTTPS path.
- Guaranteed support for all HTTP/2 or WebSocket scenarios.

## Target User and Workflow

The user is a developer inspecting traffic generated by an Android app they control. The expected workflow is:

1. Start the local app on Windows.
2. Open the dashboard in a desktop browser.
3. Generate or load the local CA certificate.
4. Configure the Android device to use the desktop machine as its Wi-Fi proxy.
5. Install and trust the local CA on the Android device when needed.
6. Launch the Android app and observe captured sessions in real time.
7. Filter sessions and inspect request and response details.

## Architecture

The system is split into four modules.

### 1. Proxy Core

Responsibilities:

- Listen for explicit proxy traffic from the Android device.
- Handle HTTP requests directly.
- Handle HTTPS `CONNECT` requests and perform MITM interception.
- Forward upstream requests to destination servers.
- Emit normalized session lifecycle events.

Constraints:

- Use Node.js and existing lower-level proxy and TLS capabilities rather than implementing TLS from scratch.
- Treat handshake failures and trust failures as first-class outcomes, not exceptional logs only.

### 2. Certificate Authority Module

Responsibilities:

- Create and persist a local CA on first run.
- Issue leaf certificates for intercepted TLS hosts.
- Expose certificate download and setup instructions through the web UI.

Constraints:

- The supported model is a user-installed CA trusted by the Android device.
- If a target app uses SSL pinning or ignores user-installed certificates, the product reports the failure but does not attempt to bypass it.

### 3. Session Store

Responsibilities:

- Persist normalized session metadata.
- Persist request and response bodies separately from hot list metadata.
- Record request timing, decode state, TLS outcome, and upstream error details.

Storage design:

- Use SQLite for session metadata and indexes.
- Store large bodies outside the main row payload to keep list queries fast.
- Mark bodies as truncated when limits are exceeded.

### 4. Web UI

Responsibilities:

- Show live session traffic.
- Allow filtering and searching.
- Render request and response detail views.
- Surface setup state and diagnostics.

## Data Flow

For an HTTPS request:

1. The Android app connects to the local proxy and sends a `CONNECT` request for the target host.
2. The proxy establishes a downstream TLS session with the Android device using a leaf certificate signed by the local CA.
3. The proxy establishes an upstream TLS session with the real destination server.
4. Decrypted HTTP messages pass through the proxy core.
5. The proxy normalizes request and response data into a session model.
6. Headers, metadata, status, timings, and storage pointers are written to SQLite.
7. Bodies are decoded when possible, persisted according to storage rules, and flagged if truncated or not previewable.
8. A live update is pushed to the browser UI.
9. The selected session can then be opened for detailed request and response inspection.

For plain HTTP, the flow is the same without certificate issuance and TLS interception.

## Technical Decisions

### Runtime and Application Shape

- Runtime: Node.js.
- Product shape: a local server process exposing both the proxy listener and the web dashboard.
- Server framework: Fastify.

### Real-Time UI Updates

- Use WebSocket or Server-Sent Events to push newly captured sessions into the UI.
- Prefer a simple append-only session stream for the MVP.

### Storage

- Use SQLite for metadata and indexes.
- Persist large bodies separately from frequently queried session fields.
- Store only a bounded amount of body content for large payloads and explicitly mark truncation.

### Body Decoding

Support in the MVP:

- Text and JSON preview.
- Common compressed payloads: `gzip`, `br`, `deflate`.
- Basic binary handling via metadata and size display.

Out of scope for MVP:

- Rich binary viewers.
- Automatic decoding for every content type and charset edge case.

### Protocol Support Boundaries

- HTTP and HTTPS are required.
- Full HTTP/2 handling is not a requirement for the first version.
- WebSocket inspection is out of scope for the first version.

## UI Design

The dashboard is a single-page application with three persistent regions.

### Left Pane: Session List

Displays a live chronological list of captured sessions with:

- HTTP method.
- Host and path.
- Status code.
- Content type indicator.
- Duration.
- Error marker when relevant.

Includes filters for:

- Host.
- Method.
- Status code.
- Text search.
- Error-only view.

### Center Pane: Session Detail

Tabbed detail area with:

- `Overview`
- `Request`
- `Response`
- `Raw`

The request view shows URL, query, headers, and body.
The response view shows status, headers, and body.
When the body is not previewable, the UI must state why.

### Right Pane: Setup and Diagnostics

Shows:

- Whether the CA exists.
- Whether the proxy listener is running.
- Basic Android setup steps.
- Recent TLS and upstream failures.
- Clear reasons when capture is incomplete.

This pane remains part of the main dashboard in the MVP so setup issues are immediately visible during first use.

## Error Handling

Errors are modeled explicitly and shown in the product.

### Setup Errors

- CA not yet generated.
- Android device not configured to use the proxy.
- Certificate not installed or not trusted.

### TLS Errors

- Handshake failed.
- Certificate rejected by client.
- TLS interception blocked by pinning or trust policy.

### Upstream Errors

- DNS resolution failure.
- Connection timeout.
- Reset by peer.
- Upstream TLS failure.

### Decode Errors

- Compression decode failed.
- Charset unknown or unsupported.
- Binary payload not previewable.

### Storage and UI Errors

- Session write failed.
- Body persistence failed.
- Live update channel disconnected.
- Body truncated due to size cap.

Each class must produce a user-visible diagnostic state and be attached to the affected session when applicable.

### Unit Tests

- Session normalization.

- Body preview eligibility.
- Compression decoding.
- Truncation rules.
- Error classification.

### Integration Tests

Cover:

### Manual Verification

Cover:

- Android device proxy configuration.
- CA generation and installation flow.
- A known-good test endpoint returning JSON.
- A compressed response.
- An intentional error case such as timeout or invalid upstream host.

## Risks and Tradeoffs

### Risk: TLS Complexity

HTTPS interception is the highest-risk part of the MVP. It introduces certificate generation, trust-chain management, and app-specific failures. The design accepts this because HTTPS visibility is a hard requirement.

### Risk: Android Trust Variability

Some apps will not trust user-installed certificates or will use SSL pinning. The product must make these failures visible and non-misleading rather than treating them as unsupported edge cases.

### Tradeoff: Use Existing Lower-Level Building Blocks

The implementation should not reimplement TLS or low-level proxy machinery from scratch in the first version. The product value is in the session model, diagnostics, and UI. This keeps the first release buildable while preserving the option to replace lower-level internals later.

### Tradeoff: Narrow Scope

The design intentionally excludes rewrite, replay, iPhone, WebSocket, and deep binary inspection so the MVP can reach a usable state quickly.

## Acceptance Criteria

The MVP is successful when all of the following are true:

- An Android device can point to the desktop machine as its Wi-Fi proxy.
- The local CA can be generated and made available for installation.
- A normal Android app that trusts the local CA can make HTTPS requests through the proxy.
- The dashboard shows those requests as sessions in real time.
- A user can inspect request and response headers and text or JSON bodies.
- Failures such as certificate trust issues, handshake problems, and upstream errors are visible in the UI with clear labels.

## Open Follow-On Work

Potential future work after the MVP:

- Replay or resend.
- Request and response rewriting.
- Better binary viewers.
- iPhone support.
- HTTP/2 and WebSocket support.
- Saved workspaces and session export.
