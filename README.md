# Android Proxy Dashboard

Android Proxy Dashboard is a local debugging tool for inspecting HTTP and HTTPS traffic from an Android device through an explicit Wi-Fi proxy.

It includes:

- a local Fastify web app for setup, session browsing, and live updates
- an explicit proxy built on `http-mitm-proxy`
- SQLite-backed session storage
- request/response body persistence
- HTTPS MITM support using a generated local CA
- a browser dashboard with live session streaming over SSE

## What It Does

The app runs two local services:

- `Dashboard` on `http://127.0.0.1:3000` by default
- `Proxy` on `0.0.0.0:8080` by default

The dashboard is local-only by default. The proxy is LAN-reachable by default so an Android device on the same Wi-Fi network can send traffic through your computer.

Once configured, you can:

- inspect captured sessions
- filter requests by host/path
- open a session to inspect headers and body previews
- download the generated CA certificate
- verify whether HTTPS interception is configured correctly

## Requirements

- Node.js installed
- An Android device on the same Wi-Fi network as your computer
- Permission to install a local CA certificate on that Android device if you want HTTPS interception

## Default Ports and Hosts

By default the app uses:

- Dashboard host: `127.0.0.1`
- Dashboard port: `3000`
- Proxy host: `0.0.0.0`
- Proxy port: `8080`
- Data directory: `.data`

That means:

- the dashboard is intended to be opened on the local machine
- the proxy can accept traffic from your Android device over the LAN

## Environment Variables

You can override the runtime defaults with these environment variables:

- `APP_PORT`
  - Dashboard HTTP port
  - Default: `3000`
- `PROXY_PORT`
  - Explicit proxy port for Android/device traffic
  - Default: `8080`
- `PROXY_HOST`
  - Network bind host for the proxy listener
  - Default: `0.0.0.0`
- `DATA_DIR`
  - Root directory for local state
  - Default: `.data`

Examples:

```powershell
$env:APP_PORT="3100"
$env:PROXY_PORT="8888"
$env:PROXY_HOST="0.0.0.0"
$env:DATA_DIR="D:\temp\android-proxy-dashboard"
npm run dev
```

## Data Layout

Inside `DATA_DIR`, the app creates:

- `sessions.db`
  - SQLite session metadata
- `certs/`
  - CA certificate files exposed to the app
- `bodies/`
  - persisted request/response bodies
- CA root files generated by `http-mitm-proxy` alongside the cert layout it requires

## Android Setup

### 1. Start the app

Run:

```bash
npm run dev
```

Open the dashboard in your desktop browser:

- [http://127.0.0.1:3000](http://127.0.0.1:3000)

### 2. Find your computer's LAN IP

On Windows PowerShell:

```powershell
ipconfig
```

Look for the IPv4 address of the network your Android device is using.

### 3. Configure the Android Wi-Fi proxy

On the Android device, connect to the same Wi-Fi network and set the proxy manually:

- Host: `your-computer-lan-ip`
- Port: `8080` by default, or your configured `PROXY_PORT`

### 4. Open the setup endpoints

The dashboard and API expose:

- `GET /api/setup`
  - returns proxy port, certificate state, and onboarding steps
- `GET /api/certificate`
  - downloads the generated CA certificate

You can use the dashboard UI or call them directly:

- [http://127.0.0.1:3000/api/setup](http://127.0.0.1:3000/api/setup)
- [http://127.0.0.1:3000/api/certificate](http://127.0.0.1:3000/api/certificate)

### 5. Install the CA on Android for HTTPS interception

If you want to inspect HTTPS traffic:

1. download the local CA from `/api/certificate`
2. install it on the Android device
3. trust it according to your Android version and device policy

Without the CA, HTTP traffic can still work, but HTTPS interception will fail.

### 6. Launch your target app

Once the Android proxy is configured, open the target app and generate traffic.

You should see sessions appear in the dashboard.

## Dashboard Overview

The dashboard has three main panes:

### Session List

Shows captured sessions with:

- method
- host/path
- status
- simple filtering

### Session Detail

When you click a session, the client fetches `/api/sessions/:id` and renders the stored session detail.

### Setup and Diagnostics

Shows setup state from `/api/setup`, including certificate availability and proxy onboarding hints.

### Live Updates

The dashboard subscribes to:

- `GET /api/events`

This is a `text/event-stream` SSE endpoint that pushes newly captured sessions to the browser.

## API Endpoints

### `GET /health`

Returns:

```json
{ "ok": true }
```

### `GET /api/sessions`

Returns the captured session list.

### `GET /api/sessions/:id`

Returns a single captured session, or `404` if it does not exist.

### `GET /api/setup`

Returns:

- proxy port
- certificate state
- Android onboarding steps

### `GET /api/certificate`

Downloads the generated CA certificate if available.

Returns `404` if the CA has not been generated yet.

### `GET /api/events`

Server-Sent Events stream for live session updates.

## Architecture Summary

High-level components:

- `src/server.ts`
  - Fastify app bootstrap
  - static assets
  - SSE endpoint
  - route registration
  - direct-run startup and shutdown
- `src/proxy/createProxyServer.ts`
  - explicit proxy lifecycle
  - HTTP and HTTPS capture
  - body persistence integration
  - event publication
- `src/storage/`
  - SQLite session storage
  - body persistence
- `src/ca/`
  - CA state inspection and path resolution
- `src/http/routes/`
  - REST endpoints
- `public/`
  - browser dashboard shell and client logic

## Limitations

- The dashboard itself is local-only by default.
- HTTPS interception only works when the client trusts the generated CA.
- Apps that use certificate pinning or custom trust logic may still fail to intercept.
- Browser-level UI behavior is only lightly automated; most verification is currently server-side and integration-test based.
- Graceful shutdown behavior is covered only indirectly, not by a dedicated shutdown/in-flight request test.
- Runtime preview decoding and the standalone `decodeBody` utility are closely aligned but not exercised by identical code paths in every case.

## Troubleshooting

### I do not see any sessions

Check:

- the app is running
- the Android device and your computer are on the same Wi-Fi network
- the Android Wi-Fi proxy host is your computer's LAN IP
- the Android Wi-Fi proxy port matches `PROXY_PORT`

### `/api/certificate` returns 404

This means the CA has not been generated yet. Trigger proxy usage so the CA material is created, then try again.

### HTTP works but HTTPS does not

Check:

- the Android device trusts the downloaded CA
- the target app does not use pinning or a restricted trust model
- the device is still using the configured Wi-Fi proxy

### Sessions appear but body previews are missing

Possible reasons:

- content type is not previewable
- payload is binary
- preview was truncated to the configured limit
- a compression/decode path did not yield previewable text

### The proxy is unreachable from Android

Check:

- you used the machine's LAN IP, not `127.0.0.1`
- local firewall rules allow inbound traffic to the proxy port
- `PROXY_HOST` is not set to loopback-only
